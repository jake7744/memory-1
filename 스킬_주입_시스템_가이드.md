# 🧠 에이전트 지식·스킬 주입 시스템 가이드 (고도화 버전)

본 문서는 Connect-AI Brain 내에서 각 에이전트(Developer, YouTube, Writer, Researcher 등)에게 새로운 능력, 템플릿, 작업 패턴, 코드 구조, 콘텐츠 포맷을 장착시키는 방법과 그 원리를 정리한 가이드입니다.

이 시스템의 목적은 에이전트가 매번 백지상태에서 작업하지 않도록 하고, 검증된 구조와 반복 가능한 패턴을 기반으로 더 빠르고 일관된 결과물을 만들게 하는 것입니다.

---

## 1. 개요 (Overview)
AI 에이전트가 특정 작업을 수행할 때마다 처음부터 구조를 판단하고, 코드를 작성하고, 문서 형식을 기획하게 두면 다음과 같은 문제가 발생합니다.
- 결과물의 품질이 매번 달라짐
- 이전에 검증한 구조를 재사용하지 못함
- 에이전트마다 작업 방식이 달라짐
- 반복 작업에 불필요한 시간 소모
- 사용자의 의도와 다른 형식의 결과물이 나올 가능성이 커짐

이를 해결하기 위해 Connect-AI Brain에서는 **패키지형 스킬 주입 시스템**을 사용합니다. 스킬 주입이란 특정 작업 패턴(예: SaaS 랜딩 페이지 구조, 유튜브 대본 포맷, SEO 블로그 템플릿, 코드 리팩토링 규칙 등)을 하나의 패키지로 만들어 에이전트의 지식 베이스에 저장해두고, 필요할 때 에이전트가 스스로 찾아서 적용하게 만드는 방식입니다.

---

## 2. 주입 시스템 핵심 폴더 구조
스킬은 다음 경로에 배치하여 에이전트별로 분리 관리합니다.
```text
내지식\40_템플릿\[에이전트명]\[스킬명]\
```

### 구조 예시:
- `내지식\40_템플릿\developer\landing-kit\` (개발용 템플릿)
- `내지식\40_템플릿\youtube\tarot-script-kit\` (유튜브 대본 구조)
- `내지식\40_템플릿\writer\seo-blog-template\` (SEO 포스팅 구조)

---

## 3. 스킬 패키지 표준 구성 요소
스킬이 완벽하게 작동하려면 해당 폴더 내에 다음 3가지 핵심 요소가 반드시 존재해야 합니다.

| 구성 요소 | 역할 |
| :--- | :--- |
| **`manifest.json`** | 스킬의 이름, 발동 조건(의도/키워드), 우선순위, 결합/충돌 조건 정의 |
| **`README.md`** | 에이전트가 실제 작업 시 따라야 할 상세 행동 매뉴얼 및 스타일 지침 |
| **`files\` 폴더** | 에이전트가 복제 및 커스터마이징하여 사용할 보일러플레이트/코드/템플릿 원본 파일 보관 |

```text
내지식\40_템플릿\[에이전트명]\[스킬명]\
│
├─ manifest.json
├─ README.md
├─ CHANGELOG.md (버전 변경 이력 관리용)
└─ files\
   ├─ blog_base.md
   ├─ Hero.tsx
   └─ config.json
```

## 4. 고도화된 manifest.json 명세서 설계

단순히 키워드(`keywords`)만 매칭하면 사용자가 *"블로그처럼 읽히는 유튜브 대본 써줘"*라고 할 때 의도와 다른 블로그 스킬이 발동되는 문제가 생깁니다. 이를 방지하기 위해 다차원적인 라우팅 규칙을 사용합니다.

### manifest.json 표준 스펙 예시
```json
{
  "name": "SEO Blog Template",
  "version": "1.0.0",
  "agent": "writer",
  "description": "검색 노출을 고려한 정보성 블로그 포스팅 작성 스킬",
  "keywords": ["블로그", "SEO", "포스팅", "검색노출", "상위노출", "글쓰기"],
  "intent": ["blog_writing", "seo_content", "information_article"],
  "avoid_keywords": ["유튜브 대본", "쇼츠", "랜딩페이지", "광고 카피", "앱 UI"],
  "base": "markdown",
  "tested_with": ["ChatGPT", "Claude", "DeepSeek"],
  "required_inputs": ["topic", "main_keyword"],
  "optional_inputs": ["target_reader", "tone", "word_count"],
  "output_type": "markdown",
  "priority": 70,
  "components": [
    "seo_title",
    "meta_description",
    "intro",
    "h2_sections",
    "faq",
    "conclusion_cta"
  ],
  "entry_file": "files/blog_base.md",
  "readme": "README.md",
  "can_combine_with": ["seo-title-kit", "faq-generator-kit"],
  "conflicts_with": ["youtube-script-template", "landing-page-copy-kit"]
}
```

### 핵심 제어 필드 정보
- **`intent`**: 사용자의 실제 의도 유형을 분류하여 정확하게 매칭합니다.
- **`avoid_keywords`**: 이 키워드가 사용자 명령에 섞여 있을 경우 오발동을 원천 회피합니다.
- **`priority`**: 동일한 키워드에 여러 스킬이 후보가 되었을 때 적용 우선순위(90~100: 핵심, 70~89: 일반, 50~69: 보조)를 결정합니다.
- **`can_combine_with` / `conflicts_with`**: 타로 대본 스킬 + 썸네일 카피 스킬처럼 조합이 가능한지 혹은 충돌하는지 정의하여 안정성을 높입니다.

## 5. README.md 표준 가이드라인 작성 양식

에이전트의 환각 현상을 억제하기 위해 README 파일은 정형화된 섹션 구조로 작성되어야 합니다.

```markdown
# [스킬 이름]

## Purpose
이 스킬의 목적을 간결하게 설명합니다.

## When to Use / When Not to Use
- **사용해야 하는 구체적인 상황**
- **절대 이 스킬을 적용해서는 안 되는 예외 상황** (예: 감성 에세이 작성 시에는 SEO 템플릿 사용 금지)

## Required & Optional Inputs
작업 전에 필요한 필수 입력값과 없어도 무방한 선택 입력값을 명시합니다.

## Output Structure
최종 결과물이 담고 있어야 하는 마크다운 레이아웃이나 코드 컴포넌트 목록을 기재합니다.

## Style Rules (Do & Don't)
- **Do**: 반드시 지켜야 할 스타일, 구조, 로직 규칙
- **Don't**: 절대 작성하지 말아야 할 금기 사항 (예: "근거 없는 임의 수치 작성 금지")

## Validation Checklist
에이전트가 결과를 출력하기 직전 최종 자가 점검(Self-Correction)을 수행할 질문지 목록
```

## 6. 스킬 발동 및 실행 워크플로우

1. **요청 및 판단:** 사용자 요청을 수집하여 매칭 후보 스킬을 선별합니다.
2. **의도 및 충돌 정제:** `intent` 일치도 판단 및 `avoid_keywords`, `conflicts_with` 검사를 통해 최적의 스킬을 라우팅합니다.
3. **입력값 검증:** `required_inputs`가 누락된 경우 합리적으로 추정(Default 값 적용)하거나, 핵심 정보인 경우 사용자에게 되물어 조달합니다.
4. **조립 및 최적화:** `entry_file`을 복제하여 `README.md` 지침에 맞춰 콘텐츠/코드를 완성합니다.
5. **자가 검증:** `Validation Checklist`를 1대1로 대응하여 오류가 없는지 최종 자가 검사 후 출력을 완료합니다.

---

## 7. 스킬 운영 및 유지 관리 모범 사례 (Best Practices)
- **모듈 단위 관리:** 하나의 스킬에 과도한 기능을 채우지 말고 `youtube\tarot-script-kit`와 `youtube\title-thumbnail-kit`처럼 작게 쪼개어 `can_combine_with`로 확장시킵니다.
- **완성된 보일러플레이트 제공:** `files/` 폴더에 들어가는 코드는 반드시 바로 컴파일 및 실행 가능한 상태(Working Code)여야 에이전트의 2차 오류를 방지할 수 있습니다.
- **버전 및 CHANGELOG 관리:** 스킬 고도화 과정에서 에이전트 호환성을 지키기 위해 버전 태그(`version`)와 `CHANGELOG.md` 작성을 강제합니다.
