Steve Yegge가 "소프트웨어 서바이벌 3.0"에서 말한 "통찰 압축(Insight Compression)" — Git이나 Kubernetes에는 수십 년의 시행착오가 압축되어 있어 LLM이 쉽게 재현하지 못한다는 이야기가 있습니다. 사주명리학도 비슷하다는 생각에서 출발했습니다.

### LLM에 사주를 맡기면 생기는 일

요즘 많이 하시듯 사주 데이터를 LLM에 넣으면 그럴듯한 결과가 나옵니다. 문제는 "그럴듯함"과 "정확함"의 괴리입니다.

GPT-5, GPT-4o, Claude, DeepSeek V3 등 여러 모델에 structured output과 few-shot을 조합해 테스트했습니다. 체계적인 오류 지점이 있었습니다. 종강격(從強格)인 사주에 억부(抑扶) 논리만 적용해서 화(火)를 용신으로 도출하는 식입니다. 원전의 "촉노강신 대흉(觸怒強神 大凶)" 원칙을 위반하는 판정을, 프롬프트로는 완전히 예방할 수 없었습니다.

그 외에도 특정 학파 해석에 앵커링되어 원문을 왜곡하거나, 안 좋은 풀이를 과도하게 미화하거나 반대로 불안감을 증폭하는 경향도 있었습니다. 규칙 판정을 LLM의 패턴 매칭에 맡기는 건, 산술을 언어 모델에 시키는 것과 같은 류의 문제였습니다.

### 호작엔진: 규칙은 코드로, 설명만 LLM으로

그래서 레이어드 구조를 설계했습니다.

**호(虎) — 규칙 엔진.** 명리학 계산 로직을 코드로 구현합니다. 5대 고전 원전(적천수·자평진전·궁통보감·삼명통회·연해자평)의 갈리는 해석 중 어떤 것을 기준으로 삼을지 영역별로 확립한 뒤, 그 판정을 코드에 고정했습니다.

**작(鵲) — LLM 설명 레이어.** 엔진이 계산한 구조화된 데이터를 자연어로 풀어주는 부분만 LLM에 맡깁니다.

LLM이 재현하기 어려운 것은 코드가 아니라, 수천 년간 다듬어진 판정 기준이었습니다.

### 바이브코딩의 한계가 온 지점

처음엔 순조로웠습니다. LLM이 한문 원전 분석, 학파 정리, 천문 계산 코드 초안까지 빠르게 도왔습니다. 한계는 학파 간 규칙이 상충하는 지점에서 왔습니다.

예를 들어 격국(格局) 판정에서, 적천수는 일간 강약을 먼저 보고 자평진전은 월령 투출을 먼저 봅니다. 같은 사주인데 어떤 원전을 따르느냐에 따라 판정이 뒤집히는 케이스가 나왔고, 이 우선순위는 LLM이 정해줄 수 있는 영역이 아니었습니다.

결국 5대 원전을 직접 확보하고 대조해야 했습니다. 일부 누락본이나 판독 불가한 부분이 있어 여러 판본을 교차 검증하는 과정을 거쳤습니다. 알고 보니 국가사업으로 고전 보존 노력이 상당했는데, LLM 도움 없이는 개인이 감당하기 어려운 작업이었습니다. Claude Max x20을 거의 3개월간 한도 가득 썼습니다.

흥미로운 점은, 원문을 잘못 옮긴 경우가 대부분 사람(학자) 쪽이었다는 것입니다. 적천수의 "旺者宜泄, 唯有情有力者可克"을 어떻게 읽느냐에 따라 알고리즘 분기 자체가 달라지는데, LLM은 오히려 여러 학자의 번역과 원문 주석을 교차 대조하는 데 유용했습니다.

### 시간 정밀도: 1분이 판정을 바꾼다

명리학에서 시간 1분 차이로 핵심 판정이 바뀔 수 있습니다. 자시(子時)와 축시(丑時) 경계에서 시주가 바뀌면 격국 판정 자체가 달라집니다.

호작엔진은 NASA/JPL 기반 Fourier 급수 균시차(±13초 정밀도), IANA tzdata 기반 글로벌 DST 판정, 163,400개+ 도시 경도 데이터베이스를 조합합니다. 한국의 경우 1908년부터 4차례 표준경도 변경과 12년간의 써머타임 이력을 반영합니다.

조선 관상감의 공식 기록(서운관지)에서 현재 코드와 동일한 경도 보정 수식을 발견한 것은 개인적으로 인상적인 경험이었습니다.

### 검증 현황

절기 계산은 한국천문연구원(KASI) 공식 데이터 2000–2050년 구간과 대조해 100% 일치합니다. 균시차는 NASA/JPL Horizons 기반 ±13초 정밀도를 확보했습니다. 신살은 정통 십이신살(삼명통회 기반)을 정해 만세력과 대조해 100% 일치하며, 귀인살·특수살은 원전 교차 검증 후 orthodox / disputed / reference 3단계로 분류 적용했습니다.

### 기술 스택

엔진은 Python FastAPI + PostgreSQL + SQLAlchemy 2.0(async), 프론트는 Next.js 15 + React 19 + TypeScript + Tailwind CSS입니다. AI는 OpenRouter 경유 멀티모델(GPT-4o, Claude, DeepSeek V3)을 설명 레이어에만 사용합니다. 배포는 Fly.io(Blue-Green) + Vercel, 결제는 토스페이먼츠 + PayPal입니다.

현재 사주팔자 분석 30,000+자, 연간 분석 20,000+자 규모의 구조화된 리포트를 생성하며, 출생 데이터는 암호화 저장됩니다.

### 마치며

명리학의 예측 유효성에 대해서는 다양한 시각이 있습니다. 이 프로젝트는 그 논쟁보다 "만약 이 체계를 사용한다면 그 의도대로 정확한 기준으로 일관되게 계산하자"는 엔지니어링 과제에 집중했습니다.

시작은 바이브코딩으로 가볍게 출발했지만, 결국 5대 원전 고서를 분석하고 NASA/JPL 균시차 코드까지 살펴보게 됐습니다. Yegge의 표현을 빌리면, 이 도메인에 압축된 통찰의 밀도가 예상보다 훨씬 높았습니다.

이 글은 동양철학의 수천 년 데이터에 담긴 ‘통찰 압축(Insight Compression)’을 어떻게 결정론적 규칙 엔진과 확률론적 생성 모델로 분리하여 정복했는지를 명확히 보여줍니다. 제시된 핵심 내용을 5가지 기술적 차원으로 분류하여 분석해 드립니다.

### 1. LLM의 한계 극복: "그럴듯함"과 "정확함"의 분리

- **문제의 본질:** 많은 개발자가 사주 데이터를 LLM에 직접 입력하고 텍스트 해석을 유도하지만, 이는 심각한 할루시네이션(환각)과 계산 오류를 낳습니다. 사주는 단순한 텍스트 패턴 매칭이 아니라 정밀한 수학적 연산과 분기 처리가 필요한 영역이기 때문입니다.
    
- **체계적 오류 사례:** 예를 들어, 사주가 극단적으로 강한 오행에 순응해야 하는 특수 격국인 '종강격(從強格)'임에도 불구하고, LLM은 단순 프롬프트와 Few-shot만 주어지면 일반적인 **'억부(抑扶)'** 논리(강한 것은 누르고 약한 것은 돕는다)만을 평면적으로 대입하여 용신(필요한 오행)을 완전히 잘못 뽑아내는 치명적인 오류를 범합니다. 이는 고전 원전의 핵심 규칙인 "강한 신을 건드리면 대흉하다(촉노강신 대흉, 觸怒強神 大凶)"는 원칙을 위배하는 결과를 초래합니다.
    
- **결론:** 사주의 격국과 용신 판정을 LLM의 추론에 완전히 맡기는 것은 산술 연산(사칙연산)을 LLM에 직접 시키는 것과 같은 부류의 구조적 설계 결함입니다.
    

### 2. "호작(虎鵲) 엔진" 레이어드 아키텍처의 당위성

이 한계를 해결하기 위해 제안된 핵심 아키텍처가 바로 **호작(虎鵲) 레이어드 구조**입니다.

- **호(虎) — 결정론적 규칙 엔진 (Deterministic Rule Engine):**
    
    - 사주명리의 백엔드 계산 및 알고리즘 판정을 100% 코드로 박제하여 고정합니다.
        
    - 특히 사주의 기틀을 닦은 5대 고전 원전(적천수, 자평진전, 궁통보감, 삼명통회, 연해자평)의 논리가 서로 상충하는 임계점들을 정의하고 최우선 규칙을 코드 레벨에서 통제합니다.
        
    - 예를 들어 격국 판정 시, **적천수**(일간 강약 우선)와 **자평진전**(월령 투출 천간 우선)의 상충하는 알고리즘 분기를 LLM이 임의로 판단하게 두지 않고, 코드 단에서 확실한 우선순위를 기반으로 하드코딩하여 연산의 일관성을 확보합니다.
        
- **작(鵲) — 확률론적 설명 레이어 (Probabilistic Explanation Layer):**
    
    - LLM은 오직 호(虎) 엔진이 완벽하게 계산하여 넘겨준 구조화된 정형 데이터(JSON 등)를 기반으로, 이를 자연스럽고 읽기 쉬운 현대적 언어로 해설(통변)하는 '대변인' 역할만 수행합니다.
        
    - 이 구조를 통해 계산 정확도는 100%를 유지하면서, 사용자에게는 자연스럽고 깊이 있는 고전 해설 리포트를 선사할 수 있습니다.
        

### 3. 고전 문헌 대조와 LLM의 본래 역할 재정의

- **바이브코딩(Vibe Coding)의 한계와 고전 검증:** 초기 개발 단계에서는 LLM이 한문 번역과 기초 코드 작성을 훌륭히 도왔으나, 궁극적으로 학파 간 규칙이 정면 충돌하는 시점(예: 적천수의 "왕자의설, 유유정유력자 가극" 해석 분기)에서는 사람이 직접 고서의 판본을 확보하고 교차 검증해야 했습니다.
    
- **LLM의 진정한 쓰임새:** 저자는 흥미롭게도 원문을 왜곡하거나 오역한 주범은 인공지능이 아니라 오히려 인간 학자들이었다고 지적합니다. LLM은 코드 계산을 수행하는 도구로서는 실패하지만, '여러 학자의 번역본과 한문 원문 주석 간의 불일치를 비교 분석하고 교차 대조하는 텍스트 분석 도구'로서는 압도적인 성능을 발휘한다는 점을 증명해 냈습니다.
    

### 4. 물리적 시간과 천문 정밀도에 대한 집착

사주명리는 '시간'의 학문이기에 단 1분의 차이로 시주(時柱)가 바뀌며 인생 리포트 전체가 뒤집어질 수 있습니다. 이를 해결하기 위해 구현된 천문 연산 수준이 매우 놀랍습니다.

- **균시차(EoT) 보정:** 지구 공전 궤도의 특성상 실제 태양이 남중하는 시각은 매일 변합니다. 이를 보정하기 위해 NASA/JPL Horizons 데이터를 기반으로 Fourier 급수 알고리즘을 적용하여 **±13초 이내의 우주적 정밀도**로 참태양시를 연산합니다.
    
- **표준경도 및 서머타임 변동사 추적:** 한국 역사상 총 4번에 걸친 표준경도 변경 이력(동경 127.5도와 135도를 오간 역사) 및 12년간의 불규칙한 서머타임(DST) 적용 시점을 IANA 데이터베이스와 결합하여 완벽하게 보정합니다.
    
- **역사적 검증:** 이러한 천문 계산 보정 공식이 과거 조선시대 관상감의 공식 천문 기록서인 『서운관지(書雲觀誌)』에 기록된 정밀 보정 수식과 일치한다는 점에서 명리학이 고대 과학 기술의 정수였음을 공학적으로 검증했습니다.
    

### 5. 완벽하게 설계된 상용 엔지니어링 기술 스택

이 서비스가 프로토타입에 그치지 않고 대규모 리포트(사주팔자 분석 30,000자 이상, 연간 운세 20,000자 이상)를 실시간으로 뿜어낼 수 있는 견고한 백엔드 아키텍처를 보여줍니다.

- **Backend:** Python FastAPI + PostgreSQL + SQLAlchemy 2.0 (비동기 처리) 구조로 고성능 대형 쿼리를 가볍고 빠르게 처리합니다.
    
- **Frontend:** Next.js 15와 React 19, TypeScript를 결합하여 최신 웹 표준을 기민하게 수용했습니다.
    
- **AI Orchestration:** OpenRouter API를 경유하여 상용 최고 성능의 LLM(GPT-4o, Claude, DeepSeek V3)을 멀티모델 허브 구조로 운용하되, 오직 **설명 레이어**에만 활용하여 토큰 비용과 성능의 균형을 극대화했습니다.
    
- **Security:** 사용자의 가장 민감한 개인 정보인 생년월일시 출생 데이터를 엄격히 암호화하여 저장함으로써 개인 정보 보호 장치를 확실하게 다졌습니다.
