## 📘 Rule Set JSON 구현 Fail-Safe Protocol v1.0
**대상:** Developer (통합 모션 그래픽 엔진 담당)
**목표:** Deep Indigo $\rightarrow$ Cream/Gold 감정적 아크 전환 시, 모든 예상치 못한 상태(Failure State)를 방지하고 안정성을 확보한다.

### 1. 시스템 레벨 실패 케이스 (System Level Failures)

| Failure State | 발생 원인 (Root Cause) | 검증 기준 및 판별 로직 (Detection Logic) | 대응 방식 (Action/Fallback) |
| :--- | :--- | :--- | :--- |
| **[F-SYS-01] Rule Set 누락** | 스크립트 단계에서 `Rule Set JSON`을 호출했으나, 해당 키가 정의되지 않은 경우. | 시스템 시작 시 로딩된 모든 콘텐츠 블록에 대해 필수 `Rule_ID` 존재 여부를 검사. (Null Check) | 텍스트 기반의 정적인 배경(Deep Indigo 유지)으로 즉시 전환하고 경고 메시지를 출력한다. 애니메이션 실행을 중지시킨다. |
| **[F-SYS-02] 데이터 타입 불일치** | JSON 내의 값이나 좌표계가 예상한 데이터 타입(예: 숫자가 아닌 문자열, 배열이 아닌 단일값)과 다를 경우. | 모든 입력 변수(`@variable`)는 API 호출 직전에 `Type Casting`을 강제하고, 실패 시 기본값을 적용한다. | 해당 모션 그래픽 블록 전체를 건너뛰고 (Skip Block), 다음 정상 로직으로 즉시 넘어간다. 오류 코드를 기록한다. |
| **[F-SYS-03] 시간 동기화 불일치** | 비디오 클립의 타임스탬프와 애니메이션 실행 타이밍(Timeline)이 오차 범위(`$\pm$ 50ms`)를 벗어날 경우. | 마스터 워크플로우 API에서 모든 모션 이벤트는 '기준 시점 시간'을 기준으로 삼고, 오차가 발생하면 즉시 지연/단축 로직을 가동한다. | 가장 가까운 *정상적인* 타이밍으로 애니메이션 속도를 보정(Pacing Adjustment)하며 진행하거나, 해당 이벤트를 무시하고 배경 전환만 수행한다. |

### 2. 비주얼 및 시각적 실패 케이스 (Visual Failures)

| Failure State | 발생 원인 (Root Cause) | 검증 기준 및 판별 로직 (Detection Logic) | 대응 방식 (Action/Fallback) |
| :--- | :--- | :--- | :--- |
| **[F-VIS-01] 색상 코드 실패** | 지정된 HEX 코드가 유효하지 않거나, Deep Indigo $\rightarrow$ Gold 전환 과정에서 목표 색상(Cream/Gold)으로의 변환이 불완전할 경우. | 모든 컬러 변경 로직은 `Color Interpolation` 함수를 사용하며, 반드시 `[Start Color] -> [End Color]` 경로가 수학적으로 검증되어야 한다. | 가장 가까운 표준 HEX 코드 (예: `#FFD700`)로 강제 대체(Hardcode Fallback)하고, 전환 과정의 *변화율*을 낮춰 부드럽게 처리한다. |
| **[F-VIS-02] 기하학적 충돌** | 겹치는 요소들이 애니메이션 과정에서 물리적으로 충돌하거나 레이어가 뒤섞이는 경우 (Layering Conflict). | 모든 모션 객체는 Z-Index 규칙을 따르며, 오버레이되는 객체의 경계값(Bounding Box)은 렌더링 직전에 검증한다. | 충돌이 감지된 요소 중 전경(Foreground) 요소의 애니메이션 우선순위를 높이고, 충돌 깊이를 시각화하는 미세한 그림자 효과를 추가하여 오류를 숨긴다. |
| **[F-VIS-03] 폰트 로딩 실패** | 특정 모션 그래픽 블록에서 요구되는 타이포그래피가 기기나 환경에 따라 정상적으로 로드되지 않는 경우. | 모든 필수 폰트는 `Web Font Stack` 방식으로 최소 2개 이상의 대체 폰트를 지정하고, 로딩 시도 후 백업 폰트 적용 여부를 체크한다. | 시스템 기본 안전 폰트(Safe System Font)로 즉시 전환하며, 해당 블록의 제목 크기(`H1`)는 5% 축소하여 레이아웃을 유지한다. |

### 3. 기능적/애니메이션 실패 케이스 (Functional Failures)

| Failure State | 발생 원인 (Root Cause) | 검증 기준 및 판별 로직 (Detection Logic) | 대응 방식 (Action/Fallback) |
| :--- | :--- | :--- | :--- |
| **[F-FUNC-01] 애니메이션 중단** | 기기 성능 문제 또는 메모리 부족으로 인해 모션이 강제로 끊기는 경우. | 프레임 단위로 GPU 및 CPU 부하를 실시간 측정하고, 특정 임계치(Threshold) 초과 시 미리 정의된 *감속/정지* 모드로 전환한다. | 가장 최근 성공했던 애니메이션 상태 (Keyframe)로 돌아가서 속도를 50% 감속시킨 후, 다음 단계로 천천히 이행하여 사용자에게 끊김을 인지시키지 않는다. |
| **[F-FUNC-02] 반복 루프 실패** | 개념 연결 노드맵 등에서 애니메이션이 끝없이 반복되거나, 목표 지점을 벗어나 순환하는 경우. | 모든 무한 루프 모션은 반드시 `Exit Condition` (종료 조건)을 가지며, 종료 조건에 도달하면 즉시 해당 블록의 트랜지션을 시작한다. | 노드맵의 중앙이나 배경으로 '수렴(Convergence)'하는 애니메이션을 강제 실행하여 루프를 멈추고 다음 내용으로 넘어간다. |
| **[F-FUNC-03] 터치/입력 불일치** | (만약 인터랙티브 요소가 추가될 경우) 사용자의 예상 동작과 시스템의 입력 값이 일치하지 않는 경우. | 최소한 of Touch Event Listener를 구현하여, 모든 상호작용 지점(Hot Zone)에 대한 기본 상태와 실패 상태를 정의한다. | 아무 반응이 없거나, 기본적인 '클릭' 애니메이션 (Scale Down -> Scale Up)을 실행하여 사용자에게 인터랙티브 요소가 작동했음을 시각적으로 피드백한다. |
