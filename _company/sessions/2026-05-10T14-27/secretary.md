# 📝 더바이브/1인 크리에이터: 콘텐츠 제작 자동화 SOP (v1.0)

**목표:** 기획부터 배포까지의 모든 과정을 표준화하고, 기술적 오류 발생 시 즉각적인 대응 절차를 확립하여 '매주 3편 영상' 목표를 안정적으로 달성합니다.

## 🚀 1단계: 전체 작업 흐름 (4-Gate Check 시스템)
모든 콘텐츠는 아래의 네 가지 Gate Check 지점을 필수로 거쳐야 합니다.

1.  **[기획]**: 시장 트렌드 및 감성 아크 확정 (Researcher → Writer)
2.  **[스크립트]**: 구조화된 스토리라인 완성 (Writer $\rightarrow$ Developer/Reviewer)
3.  **[시각화]**: 기술 구현 가능성 검토 및 에셋 제작 (Developer & Designer)
4.  **[배포]**: 최종 콘텐츠 확정 및 채널 최적화 (All Agents → CEO Review)

## ✅ 2단계: 단계별 상세 워크플로우 및 필수 액션

| 단계 | 담당 에이전트 | 주요 산출물 | 핵심 프로세스 및 체크리스트 |
| :--- | :--- | :--- | :--- |
| **1. 기획 (Plan)** | Researcher | 주제 선정, 키워드 리포트, 컨셉 정의 | - **[필수]** 타깃 청중(20~50대)의 공감 포인트 도출.<br>- **[핵심]** '무의식/타로' 관련 최신 트렌드와 시장 가치를 결합하여 1차 주제 확정. |
| **2. 스크립트 (Script)** | Writer | 스토리라인 초안, 대본(스크립트), 제목 후보군 | - **[필수]** 도입부 후킹(Hook)과 CTA(Call To Action) 구조 명시.<br>- **[구조]** Deep Indigo $\rightarrow$ Cream Gold의 감성적 흐름을 스크립트에 기술적으로 녹여낼 지점 지정. |
| **3. 시각화 (Visualize)** | Developer & Designer | 애니메이션 스펙, 와이어프레임, 최종 에셋 세트 | - **[기술]** Opening/Closing Sequence Schema 기반으로 Mock API 인터랙티브 프로토타입 제작.<br>- **[검수]** 모든 애니메이션 요소의 타이밍 및 HEX 코드를 명세화하고 개발 가능 여부 검증. |
| **4. 배포 (Deploy)** | All Agents | 최종 영상 파일, 제목/설명란 세트, 플레이리스트 | - **[최적화]** 유튜브 알고리즘에 최적화된 제목(클릭률/검색) 및 재시청 유도형 설명란 작성.<br>- **[점검]** 배포 전 모든 콘텐츠가 'Gate Check'를 통과했는지 최종 검토. |

## 🚨 3단계: 필수 Gate Check (Critical Validation Points)
다음 세 가지는 각 단계 완료 후 반드시 확인해야 할 최소한의 검수 항목입니다. 이 중 하나라도 미흡하면 다음 단계로 진행 불가합니다.

1.  **💡 감성 아크 유무 검증:** 콘텐츠 전반에 걸쳐 'Deep Indigo'에서 'Cream Gold'로의 명확하고 자연스러운 감정적 흐름이 기술적으로 구현될 수 있는지 확인해야 합니다. (감각적인 스토리텔링과 시각적 변화가 필수)
2.  **⚙️ 기술 구현 가능성 검증:** 기획된 모든 애니메이션, 전환 효과, 상호작용(Interaction)은 현재 개발 스펙 및 보유 에셋 내에서 **기술적으로 100% 구현 가능한지**를 Developer가 최종 확정해야 합니다. (Mock API 기반의 안정화 우선)
3.  **🔎 목표 달성 명시 검증:** 해당 영상이 '구독자 증가' 또는 '재시청 유도' 중 어떤 단일 목표를 추구하는지, 그리고 그에 맞는 CTA(Call To Action)가 스크립트와 시각 요소 모두에서 반복적으로 노출되는지 확인해야 합니다.

## ⚠️ 4단계: 에러 핸들링 및 안정화 지침 (Error Handling Protocol)
시스템 오류나 작업 병목 현상이 발생했을 때의 대응 매뉴얼입니다.

*   **[LLM 호출 실패 시]:**
    *   ❌ **절대 원인 파악에 시간을 낭비하지 마세요.**
    *   ✅ 즉시 모델 오케스트레이션 모달에서 Context Length를 점검하고, 더 작은 LLM 모델을 우선 선택하여 작업 안정성을 확보합니다. (Context Length 최소 8192 이상 유지)
*   **[메모리/자원 부족 시]:**
    *   ✅ 가장 비즈니스 가치가 높은 단일 작업을 최우선으로 재설정하고, 불필요한 백그라운드 작업을 일시 중지합니다.
*   **[병목 지점 발생 시 (Bottleneck)]:**
    *   ✅ 개발팀/디자인팀과 협업하여 '최소 기능 제품(MVP)' 관점으로 범위를 축소합니다. 완벽함보다는 **'일단 작동하는 최소한의 결과물'**을 빠르게 확보하고, 추후 개선하는 방식으로 회복탄력성을 높입니다.
